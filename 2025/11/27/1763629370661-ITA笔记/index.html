发表于 2025-11-27 标签： 式样书做成

ITA测试：将参照资料转化为“自己能用”的式样

1. 自己先吃透，再去问与建议

  • 基本原则：
    只有自己先将一件事情尽可能地了解透彻之后，才有资格去问别人问题，才有资格向别人提供建议和想法。
  • 对应行动：
    在做ITA测试式样书之前，要先把资料、规格、现有程序行为，都转化成自己能理解、能操作的形式（图、表、例子、请求样本等）。

2. 项目开始时全局统括 mock 的重要性

  • 全局 mock 的目标：
    在一个项目开始时，做一个统括全局的、可以在多个机能之间跳转页面的 mock，是非常重要的。
  • mock 中应包含的内容：
    • 客户端： 页面之间的跳转关系、主要画面要素。
    • 后端 API 请求：
      • 请求内容
      • 请求格式
    • 后端 API 响应：
      • 响应内容
      • 响应格式
    • 常驻 batch：
      • 各种正常情况对 DB 的更新
      • 各种异常情况对 DB 的更新

3. ITA以上结合测试所需的辅助工具准备

  • 需要的工具：
    • シミュレーター（模拟浏览器送信页面等）
    • Postman 等 API 发送工具
  • 准备的意义：
    ITA 以上的结合测试，需要借助这些发送请求的辅助工具，才能更好、更高效地进行后续作业。
  • 时间点要求：
    在正式着手 ITA 测试式样书之前，就要把这些前提事项准备好（环境、工具、账号、基础数据等）。

4. 制作结合测试式样书（ITA，ITB）的步骤概述

4.1 前提准备

  • 前提准备1：
    使用模拟浏览器送信页面，或者 Postman，配合程序实际执行的过程和结果，来制作测试式样书。
    • 一边实际发送请求，一边确认响应和 DB 变更
    • 把“事实行为”整理成测试观点和测试步骤

4.2 步骤一：制作ケース一览

  • 目的：
    简单介绍每个 case 的测试内容，形成一个“测试全体地图”。
  • 内容示例：
    • Case ID
    • 测试标题
    • 涉及机能（画面 / API / batch）
    • 主要测试目的（正常系 / 异常系 / 边界值等）

4.3 步骤二：为每条 case 写确认观点

  • 确认观点示例：
    • 响应相关：
      • 响应的 HTTP Status
      • 响应 JSON / XML 的各项值
    • DB 相关：
      • DB 登录值是否正确
      • DB 更新值是否符合预期
      • 记录条数、状态值、时间戳等是否正确

4.4 步骤三：写 case 实施的条件与步骤（概要）

  • 实施条件：
    • 需要事先准备的前提数据
    • 必须满足的业务状态（例如：订单已创建但未支付等）
  • 实施步骤（概要版）：
    将为了满足测试观点所需的主要条件，按顺序列出，并对实施步骤做简要描述。

4.5 步骤四：详细写出条件、数据与实施步骤（详细版）

  • 详细条件与数据：
    • 请求体中各项目的值（字段、类型、具体示例值）
    • DB 登录值、更新值的具体内容（表名、字段名、预期值）
  • 详细实施步骤：
    • 具体操作步骤（在模拟画面/浏览器/工具上的操作）
    • 请求发送方式（URL、Method、Headers、Body 等）
    • 预期结果检查方法（从响应、日志、DB 等处如何确认）
